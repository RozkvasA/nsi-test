# Этап 12: упрощение отображения данных и режимы чтения

Этот документ применяется вместе с:

```text
docs/NSI_MODULE_TARGET_SPEC.md
docs/NSI_OVERVIEW_USABILITY_FIX.md
docs/NSI_STAGE10_ENGINEERING_STRUCTURE_AND_CARD_UX.md
docs/NSI_STAGE11_OBJECT_TREE_ENGINEERING_VIEW.md
```

## 1. Назначение этапа

Этап 12 нужен не для добавления новых сущностей ради новых сущностей, а для снижения визуальной перегрузки интерфейса и исправления практической логики отображения данных.

Текущая модель становится логически правильнее: объект имеет Помещения и Системы, система является инженерным контуром, оборудование является физической единицей размещения.

Но интерфейс не должен заставлять пользователя видеть всю сложность модели одновременно.

Цель: сделать отображение данных спокойным, очевидным и практичным.

## 2. Практическая проблема

Пользователь не должен ощущать, что дерево объектов это перегруженная техническая схема.

Если в одной строке или на одном экране одновременно показывать:

```text
помещения
системы
наследуемые системы
оборудование
состав оборудования
предупреждения
служебные признаки
```

интерфейс становится тяжелым и неочевидным.

Правильный подход: сначала показывать структуру, а детали раскрывать постепенно.

## 3. Принцип прогрессивного раскрытия

По умолчанию дерево должно показывать только навигационный минимум:

```text
название
тип коротко
количество вложенных помещений
количество систем
критичное предупреждение, если есть
```

Остальное показывать только после раскрытия соответствующей папки или карточки.

Нельзя в каждой строке постоянно показывать все зависимости.

## 4. Режимы отображения дерева

Для Дерева объектов нужны два режима отображения:

```text
Компактный
Подробный
```

Компактный режим используется по умолчанию.

В компактном режиме:

```text
показывать короткие строки
не показывать длинные summary
не показывать состав оборудования до раскрытия системы
не показывать полный список наследуемых систем в каждой дочерней папке
показывать только счетчик наследуемых систем или короткий бейдж
```

Подробный режим:

```text
показывать больше счетчиков
показывать наследуемые системы в списке
показывать больше диагностик
показывать состав инженерной структуры глубже
```

Если нет времени делать переключатель, нужно хотя бы сделать поведение как компактный режим по умолчанию.

## 5. Системы в дереве: не перегружать дочерние уровни

Наследуемые системы не должны превращать каждую дочернюю папку Системы в огромный список одинаковых элементов.

В компактном режиме для дочернего объекта показывать так:

```text
Системы · сист. 0 · наслед. 4
```

или внутри папки:

```text
наследуется 4 системы
```

Полный список наследуемых систем показывать только по раскрытию или в подробном режиме.

Собственные системы показывать всегда, потому что они заведены именно на этом уровне.

## 6. Система как контур, оборудование как наполнение

В дереве система должна выглядеть как контур, а не как физическая единица.

Строка системы должна показывать:

```text
название системы
область действия коротко
количество оборудования
предупреждение нет оборудования, если пустая
```

Не показывать в строке системы:

```text
место размещения
placementObjectId
длинный список помещений
длинный список оборудования
```

Оборудование показывать только внутри раскрытой системы или группы Самостоятельное оборудование.

## 7. Оборудование: тип, модель и единица оборудования

Нужно различать агрегированный уровень оборудования и конкретную единицу оборудования.

Агрегированный уровень может описывать:

```text
вид оборудования
модель или типоисполнение
группу однотипных единиц
```

На этом уровне количество допустимо.

Пример:

```text
Кондиционер Open space СОТА · ед. 4
```

Но при раскрытии такой строки должны быть видны 4 конкретные единицы оборудования:

```text
Кондиционер Open space СОТА
├── AC-SOTA-OS-001 · размещение: Open space
├── AC-SOTA-OS-002 · размещение: Open space
├── AC-SOTA-OS-003 · размещение: Open space
└── AC-SOTA-OS-004 · размещение: Open space
```

Каждая единица оборудования должна быть отдельной строкой и иметь свое место размещения.

Для конкретной единицы оборудования количество не должно быть основным редактируемым смысловым полем. Единица оборудования уникальна, поэтому ее количество по смыслу равно 1.

Если в текущей модели поле quantity остается технически обязательным, в UI для единицы оборудования его нужно скрывать, фиксировать как 1 или показывать только в служебных сведениях.

Количество показывать и редактировать только для агрегированных уровней: вид, группа, типоисполнение, партия однотипного оборудования.

## 8. Родительское оборудование показывать только когда нужно

Если у оборудования нет родительского оборудования, не нужно постоянно показывать пустое поле Родительское оборудование.

Правило:

```text
если parentEquipmentId есть, показывать родительское оборудование
если parentEquipmentId нет, поле не показывать в основной части карточки
если пользователь выбирает действие Добавить составную часть или Выбрать родителя, тогда показывать выбор родителя
```

Пустые поля, которые не помогают понять объект, не должны захламлять карточку.

## 9. Описание объекта и нейтральные пространственные уровни

Не весь БЦ заполнен офисами. Часть помещений может быть не арендована, часть может быть зоной, крылом, блоком, МОП или технической зоной.

Поэтому дерево не должно навязывать только структуру БЦ -> этаж -> офис.

Более универсальная логика:

```text
БЦ
этаж
крыло
зона
блок
офис
помещение
техническое помещение
```

Название строки должно быть нейтральным и чистым:

```text
4 этаж
Левое крыло
Open space
Серверная
```

Если нужно пояснить, что Левое крыло сейчас занято офисом СОТА, это должно быть не частью названия через тире, а описанием объекта.

Пример:

```text
Название: Левое крыло
Описание: Офис СОТА
```

В строке дерева можно показывать маленькую подпись из описания, если она короткая и не перегружает строку.

Если в модели еще нет поля description у InfrastructureObject, нужно добавить его или использовать отдельный параметр description в parameters. Предпочтительно добавить явное поле description.

## 10. Оборудование в помещениях: показывать только релевантное помещению

Если система относится к БЦ и имеет 10 единиц оборудования, это не означает, что все 10 единиц нужно показывать в каждом этаже, офисе и помещении.

В конкретном помещении отображаются только те единицы оборудования, у которых placementObjectId указывает на это помещение или на этот объект размещения.

Пример:

```text
Система вентиляции относится к БЦ.
В Open space размещен кондиционер AC-SOTA-OS-001.
В Серверной размещен кондиционер AC-SOTA-SRV-001.
```

В карточке Open space показывать только AC-SOTA-OS-001.

В карточке Серверной показывать только AC-SOTA-SRV-001.

Не показывать все оборудование всей системы в каждом помещении.

## 11. Система уровня этажа или БЦ может иметь оборудование в отдельных помещениях

Система может быть создана на уровне БЦ или этажа, но ее единицы оборудования размещаются точечно.

Пример:

```text
4 этаж
└── Системы
    └── Кондиционирование 4 этажа
        └── Кондиционеры
            ├── AC-SOTA-OS-001 · Open space
            ├── AC-SOTA-OS-002 · Open space
            ├── AC-SOTA-SRV-001 · Серверная
            └── AC-MOP-001 · МОП 4 этажа
```

В карточке 4 этажа можно показать общую систему.

В карточке конкретного помещения показывать только единицы оборудования, размещенные в этом помещении.

## 12. Тестовые данные для 4 этажа и Офиса СОТА

Нужно обновить демо-данные так, чтобы на 4 этаже и в зоне Офиса СОТА были реалистичные собственные системы и оборудование.

Добавить или проверить:

```text
4 этаж
└── Помещения
    └── Левое крыло
        описание: Офис СОТА
        └── Помещения
            ├── Open space
            ├── Серверная
            ├── Переговорная
            ├── Кухня
            └── Коридор
```

В Офисе СОТА или Левом крыле должны быть собственные системы, например:

```text
Кондиционирование Офиса СОТА
Локальная СКС Офиса СОТА
Освещение Офиса СОТА
```

Для этих систем добавить единицы оборудования с placementObjectId конкретных помещений.

Пример для Open space:

```text
Кондиционер Open space СОТА
├── AC-SOTA-OS-001 · Open space
├── AC-SOTA-OS-002 · Open space
├── AC-SOTA-OS-003 · Open space
└── AC-SOTA-OS-004 · Open space
```

Также добавить самостоятельное оборудование без системы, но с placementObjectId:

```text
Серверный шкаф SRV-SOTA-001 · Серверная
Кофемашина COF-SOTA-001 · Кухня
```

Каждая единица оборудования должна быть привязана к конкретному помещению.

В карточке каждого помещения должны отображаться только его единицы оборудования.

## 13. Оборудование: показывать только там, где пользователь ожидает

Оборудование не должно засорять уровни помещения, если пользователь просто смотрит структуру помещений.

Правильные места для оборудования:

```text
внутри раскрытой системы
внутри группы Самостоятельное оборудование
в карточке помещения в таблице Оборудование в помещении
в карточке оборудования
```

Нельзя дублировать одно и то же оборудование слишком много раз в дереве.

Если оборудование отображается как часть системы, в помещении его достаточно показывать в карточке помещения, а не обязательно раздувать дерево помещений.

## 14. Папки Системы и Помещения

Папки Системы и Помещения должны быть понятными и спокойными.

Папка Помещения:

```text
показывает вложенные помещения и объекты размещения
не показывает оборудование прямо в строке
```

Папка Системы:

```text
показывает собственные системы
показывает краткий признак наследуемых систем
показывает Самостоятельное оборудование отдельной группой только если оно есть
```

Если папка пустая, показывать коротко:

```text
нет помещений
нет систем
```

## 15. Карточка как место деталей

Дерево должно быть навигатором, а не местом полного анализа.

Подробности должны уходить в правую карточку:

```text
список оборудования помещения
полный список наследуемых систем
детали области действия системы
состав оборудования
служебные идентификаторы
диагностики
описание объекта
```

В дереве оставлять только короткие признаки и счетчики.

## 16. Визуальная плотность

Интерфейс должен быть менее шумным:

```text
меньше текста в строках
меньше бейджей одновременно
меньше повторяющихся подписей
меньше ярких цветов
больше воздуха между смысловыми группами
```

При этом строки дерева не должны стать огромными карточками.

Нужен баланс: компактная строка + понятная иерархия.

## 17. Адаптивность

На большом мониторе можно показывать дерево и карточку рядом.

На ноутбуке:

```text
дерево уже
карточка шире
строки дерева максимально компактные
без горизонтального скролла всей страницы
```

На iPad/tablet:

```text
дерево и карточка не съезжают
допускается вертикальная компоновка дерево сверху, карточка ниже
действия не должны вылезать за экран
длинные summary скрываются или сокращаются
```

## 18. Что не делать

Нельзя:

```text
добавлять новые сущности без необходимости
возвращать плоское дерево без папок Системы / Помещения
дублировать наследуемые системы как реальные системы дочернего объекта
показывать оборудование одновременно везде
показывать систему как физическое место
заставлять пользователя читать длинные строки
показывать quantity как основной смысл единицы оборудования
показывать все оборудование системы в каждом помещении
ломать Обзор объектов
```

## 19. Проверка этапа

После реализации проверить:

```text
1. Обзор объектов не изменился.
2. Дерево объектов стало визуально проще.
3. В компактном режиме строки дерева короткие.
4. Нет ощущения, что в каждой строке показана вся база данных.
5. Наследуемые системы не раздувают дочерние уровни.
6. Система показывает область действия и количество оборудования.
7. Пустая система явно помечена как нет оборудования.
8. Оборудование видно внутри раскрытой системы или в карточке помещения.
9. Самостоятельное оборудование видно отдельной группой только если оно есть.
10. Длинные списки и диагностики перенесены в карточки.
11. Кондиционер Open space СОТА с количеством 4 раскрывается в 4 конкретные единицы оборудования.
12. Каждая единица оборудования имеет placementObjectId конкретного помещения.
13. У единицы оборудования quantity не выглядит как основной редактируемый смысл.
14. Если parentEquipmentId нет, поле Родительское оборудование не отображается в основной части карточки.
15. В карточке объекта есть описание или короткая подпись.
16. Левое крыло может иметь описание Офис СОТА без названия вида 4 этаж - Офис СОТА.
17. В карточке Open space отображается только оборудование Open space.
18. В карточке Серверной отображается только оборудование Серверной.
19. Оборудование всей системы не показывается в каждом помещении.
20. На ноутбуке нет горизонтального скролла всей страницы.
21. На iPad/tablet интерфейс не разваливается.
22. Пользователь может понять структуру БЦ за 10 секунд без чтения длинных подписей.
```
