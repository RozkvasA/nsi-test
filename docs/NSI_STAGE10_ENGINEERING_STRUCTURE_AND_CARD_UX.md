# Этап 10: инженерная структура, карточки и связи

Этот документ применяется вместе с docs/NSI_MODULE_TARGET_SPEC.md и docs/NSI_OVERVIEW_USABILITY_FIX.md.

## 1. Назначение этапа

Этап 10 нужен для приведения карточек элементов, связей, систем и оборудования к понятной эксплуатационной логике.

Главная идея: пользователь работает не с техническими id и checkbox-простынями, а с понятными карточками, таблицами связей и поисковым добавлением элементов.

## 2. Русские подписи параметров

В карточках элементов все подписи должны быть на русском языке.

Нельзя показывать пользователю технические названия как основные подписи:

```text
systemId
placementObjectId
typeId
parentId
rootObjectId
scopeObjectIds
linkedRoomIds
parentEquipmentId
```

Правильно:

```text
Система
Место размещения
Вид элемента
Родительский объект
Корневой объект
Область действия
Связанные помещения
Родительское оборудование
```

Технический id можно показывать только в отдельном служебном блоке, если он нужен для отладки, и с русской подписью:

```text
Идентификатор системы
Идентификатор оборудования
Идентификатор объекта размещения
```

В обычном режиме карточки технические id лучше скрывать.

## 3. Убрать Показать пустые

Переключатель Показать пустые в карточке элемента нужно убрать.

Поля должны отображаться по нормальной логике:

```text
основные обязательные поля показываются всегда
пустые необязательные поля не создают визуальный шум
пустые обязательные поля подсвечиваются предупреждением
дополнительные параметры вида можно показывать компактным блоком или раскрываемой секцией
```

Нельзя заставлять пользователя вручную управлять тем, видеть ему пустые поля или нет.

## 4. Вкладка Параметры без бокового меню Основные / Связи / Прочие

В карточках объектов текущий боковой переключатель Основные / Связи / Прочие нужно убрать или заменить на компактный второй уровень внутри вкладки Параметры.

Главные вкладки карточки остаются верхними:

```text
Параметры
Документы
Заметки
Карта
Обслуживание
```

Внутри вкладки Параметры допускается один из двух вариантов:

### Вариант А. Одна страница

```text
Основные данные
Иерархия и размещение
Параметры по виду
Связанные системы
Оборудование в помещении
Служебные сведения, если нужны
```

### Вариант Б. Маленькие под-вкладки второго уровня

```text
Основные
Параметры вида
Связанные элементы
Служебное
```

Но это не должно выглядеть как отдельное левое меню внутри карточки.

## 5. Связи как таблицы и поисковое добавление

Текущие большие checkbox-списки связей нужно заменить на понятные таблицы или списки выбранных связей.

Правильный паттерн:

```text
в состоянии покоя пользователь видит только уже выбранные связи в таблице или списке
рядом есть кнопка Добавить / Прикрепить / Связать
поиск и список выбора появляются только после нажатия на кнопку или при вводе поискового запроса
после выбора элемент добавляется в таблицу связей
```

Нельзя в состоянии покоя показывать огромный список всех возможных объектов с чекбоксами.

## 6. Связь помещение ↔ оборудование

Под двусторонней связью понимается отображение одной и той же связи в карточках обеих сущностей.

Правило:

```text
одно помещение может содержать много единиц оборудования
одна единица оборудования может находиться только в одном помещении или объекте размещения
```

В карточке помещения нужно показывать таблицу оборудования, размещенного в этом помещении.

Пример колонок:

```text
Наименование
Вид оборудования
Система, если есть
Количество
Единица измерения
Действия
```

В карточке помещения должна быть кнопка:

```text
Добавить оборудование
```

После нажатия открывается поисковый выбор оборудования. Пользователь ищет единицу оборудования и выбирает ее. Выбранная единица появляется в таблице оборудования помещения.

В карточке оборудования нужно показывать место размещения:

```text
Помещение или объект размещения
```

Там должна быть кнопка:

```text
Прикрепить к помещению
```

или, если помещение уже задано:

```text
Изменить помещение
```

После выбора помещения у оборудования меняется placementObjectId. В карточке помещения это оборудование появляется автоматически.

## 7. Связь помещение ↔ системы

Помещение может быть связано с несколькими системами.

В карточке помещения нужно показывать таблицу связанных систем.

Пример колонок:

```text
Наименование системы
Вид системы
Область действия
Количество оборудования
Действия
```

В карточке помещения должна быть кнопка:

```text
Добавить систему
```

Поиск систем открывается только после нажатия на кнопку или ввода запроса.

В карточке системы нужно показывать связанные помещения таблицей, а не огромным checkbox-списком.

## 8. Единая инженерная структура

Нужен единый рабочий список или дерево для систем и оборудования. Его можно назвать:

```text
Инженерная структура
```

или

```text
Системы и оборудование
```

Название должно быть нейтральным и понятным. Это не отдельный старый раздел Системы из левого меню, а логика отображения внутри дерева/карточек или будущего рабочего раздела.

Инженерная структура должна поддерживать два сценария:

```text
оборудование как состав системы
оборудование как самостоятельный элемент вне системы
```

## 9. Дерево систем и оборудования

Для систем должно быть свое дерево инженерной структуры.

Пример для системы пожарной сигнализации:

```text
Система пожарной сигнализации
└── Пожарная сигнализация
    └── Извещатели
        └── Извещатели дымовые
            └── Извещатель пожарный дымовой ИП 212
                ├── ИП-212-0001
                ├── ИП-212-0002
                └── ИП-212-0003
```

Смысл уровней:

```text
система
подсистема или функциональная группа
вид оборудования
модель или типоисполнение оборудования
единица оборудования с уникальным id
составная часть оборудования, если оборудование комплексное
```

Рядом с системами могут быть самостоятельные виды оборудования с единицами оборудования, если они не входят в конкретную систему.

Пример:

```text
Инженерная структура
├── Система пожарной сигнализации
│   └── Извещатели
│       └── Извещатель пожарный дымовой ИП 212
│           ├── ИП-212-0001
│           └── ИП-212-0002
├── Вентиляция и кондиционирование
│   └── Вентиляционная установка
│       └── ВУ-4-001
└── Самостоятельное оборудование
    └── Серверный шкаф
        └── SRV-SOTA-001
```

## 10. Оборудование может быть комплексным или одиночным

Оборудование может быть:

```text
одиночной единицей
типом оборудования с несколькими единицами
комплексным оборудованием с составными частями
частью системы
самостоятельным оборудованием вне системы
```

Если оборудование является частью системы, оно связано с системой.

Если оборудование самостоятельное, оно может не иметь системы, но все равно должно иметь:

```text
вид оборудования
единицу учета
место размещения
уникальный id или инвентарный номер, если это единица оборудования
```

Текущая модель может хранить это через systemId = null или через специальный верхний узел Самостоятельное оборудование. Конкретное техническое решение выбрать при реализации, но пользовательская логика должна быть именно такой.

## 11. Карточка системы

В карточке системы вкладка Оборудование должна быть не просто плоским списком, а деревом состава системы.

В состоянии покоя показывается дерево состава:

```text
функциональные группы
виды оборудования
единицы оборудования
составные части
```

Кнопки:

```text
Добавить группу
Добавить вид оборудования
Добавить единицу оборудования
Добавить составную часть
```

Если пока это слишком крупно для одного шага, минимально нужно заменить плоский список оборудования на компактное дерево с parentEquipmentId.

## 12. Карточка оборудования

Карточка оборудования должна показывать:

```text
Основные данные
Система, если оборудование входит в систему
Место размещения
Родительское оборудование, если есть
Дочерние элементы или составные части
Параметры по виду оборудования
```

Если оборудование самостоятельное, поле Система должно показывать значение:

```text
Не входит в систему
```

При этом не должно быть ошибки только из-за отсутствия systemId, если оборудование осознанно заведено как самостоятельное.

## 13. Поиск при добавлении связей

Для добавления связей использовать поисковый выбор.

Правильный паттерн:

```text
кнопка Добавить оборудование
открывается строка поиска
пользователь вводит часть названия, id или вида
ниже появляется список совпадений
пользователь выбирает элемент
выбранный элемент добавляется в таблицу
поиск закрывается или очищается
```

В состоянии покоя поисковый список не виден.

Это правило относится к:

```text
добавлению оборудования в помещение
прикреплению оборудования к помещению
добавлению системы к помещению
добавлению помещения к системе
выбору родительского оборудования
добавлению единиц оборудования в систему
```

## 14. Что не делать

Нельзя:

```text
показывать пользователю технические поля как основные подписи
оставлять ползунок Показать пустые
держать Основные / Связи / Прочие как громоздкое боковое меню в карточке
показывать огромные checkbox-списки связей в состоянии покоя
считать оборудование всегда только частью системы
считать оборудование всегда самостоятельным
показывать оборудование в Обзоре объектов как отдельный блок последнего уровня
```

## 15. Проверка этапа

После реализации нужно проверить:

```text
1. В карточках нет технических подписей systemId, placementObjectId, typeId как основных labels.
2. Ползунок Показать пустые отсутствует.
3. Вкладка Параметры стала понятной: без бокового меню Основные / Связи / Прочие или с аккуратными под-вкладками второго уровня.
4. В карточке помещения есть таблица оборудования помещения.
5. Добавить оборудование открывает поиск только после действия пользователя.
6. В карточке оборудования видно помещение размещения.
7. Изменение помещения у оборудования отражается в карточке помещения.
8. В карточке помещения есть таблица связанных систем.
9. В карточке системы связанные помещения отображаются таблицей, а не огромным checkbox-списком.
10. В карточке системы оборудование отображается деревом состава.
11. Оборудование может быть самостоятельным или входить в систему.
12. Оборудование не отображается отдельным блоком в Обзоре объектов.
```
