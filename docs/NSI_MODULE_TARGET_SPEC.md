
# НСИ: целевое описание и ТЗ для разработки

## 0. Правило для ИИ

Перед любой задачей по этому проекту сначала прочитай этот файл. Решение должно соответствовать требованиям ниже. Если задача касается дерева объектов, видов объектов, помещений, систем, оборудования, техкарт, справочников, параметров или связей, этот документ является основным источником требований.

Базовая сущность проекта: объект инфраструктуры. Модуль должен одинаково подходить для БЦ, ЖК, отеля, склада, производственной площадки, филиала, паркинга и нестандартного объекта.

Скриншоты используются только как ориентир интерфейса: слева меню, в центре рабочая область выбранного раздела, справа карточка выбранного элемента в разделах, где карточка нужна. Содержимое деревьев со скриншотов копировать нельзя.

## 1. Назначение

НСИ нужен для структурированного хранения исходных данных. Эти данные дальше используют другие модули.

НСИ не рассчитывает бюджет, не строит план работ и не формирует регламенты автоматически. Он только хранит структуру, параметры и связи.

В НСИ должны храниться: объекты инфраструктуры, обзор объектов, дерево объектов учета, дерево видов объектов, помещения, системы, оборудование, технологические карты, справочники, параметры и связи между сущностями.

Техкарты находятся в НСИ, но создаются пользователем вручную и обязательно привязываются к помещению, системе, оборудованию или виду объекта, для которого регламент применим.

Основной пользователь может быть любым: администратор, инженер, специалист эксплуатации, сотрудник УК, руководитель объекта или иной пользователь. Поэтому интерфейс должен быть универсальным, понятным и пригодным для заполнения с нуля.

## 2. Общий интерфейс

Интерфейс строится по схеме:

```text
верхняя служебная панель
левое меню НСИ
центральная рабочая область выбранного раздела
правая карточка выбранного элемента, если раздел работает с выбранной строкой
```

Разделы в левом меню:

```text
Обзор объектов
Дерево объектов
Дерево видов объектов
Техкарты
Справочники
```

Первый экран: Обзор объектов.

Обзор объектов показывает краткое содержание НСИ без полной технической иерархии. Дерево объектов показывает полную редактируемую иерархию.

В разделе Дерево объектов в центральной области нужны поиск, сортировка, свернутые строки по умолчанию, цифровая сводка в строке, раскрытие веток, выбор элемента, контекстные действия, drag and drop, действия Создать, Снять с учета, Копировать.

В правой карточке объектов и видов объектов нужны вкладки: Параметры, Документы, Заметки, Карта, Обслуживание. На первом этапе обязательно реализовать вкладку Параметры. Остальные вкладки можно оставить заглушками.

Для карточки технологической карты используется отдельный набор вкладок: Параметры, Операции, Персонал, Материалы, СИЗ.

Редактирование лучше делать в правой карточке, а не через модальное окно поверх всего экрана.

## 2.1. Правила интерфейса по референс-скринам

Референс-скрины задают визуальный паттерн интерфейса, но не задают жесткую предметную модель и не должны копироваться как готовые данные. Нельзя переносить из скриншотов конкретные деревья, названия объектов, состав старого бюджетного меню или частную структуру одного заказчика.

Основной экран НСИ должен выглядеть как рабочее место с тремя зонами:

```text
верхняя служебная панель
левое меню разделов НСИ
рабочая область выбранного раздела
```

Верхняя служебная панель содержит хлебные крошки текущего раздела слева и пользовательский блок справа. Хлебные крошки должны показывать контекст, например НСИ -> Обзор объектов, НСИ -> Дерево объектов или НСИ -> Техкарты. Панель не должна забирать много высоты.

В верхней служебной панели должен быть компактный переключатель тестовых данных. Он не должен занимать много места и не должен превращаться в отдельный экран настройки. Логика простая: тестовые данные выключены — пользователь видит пустой НСИ и может заполнять его с нуля; тестовые данные включены — загружается реалистичный демо-набор, начиная с БЦ Lucky.

Левое меню должно оставаться слева на desktop и tablet-ширинах, включая iPad. Нельзя переносить основную навигацию НСИ наверх в виде горизонтальных карточек разделов. Верхняя панель не заменяет левое меню.

Левое меню должно быть сворачиваемым. В раскрытом состоянии показываются иконка, название раздела и короткое описание или подпись. В свернутом состоянии остается узкая вертикальная панель с иконками и подсказками. Свертывание не должно менять выбранный раздел и не должно сбрасывать состояние дерева или карточки.

Кнопка сворачивания левого меню должна быть максимально компактной. В ней должна быть только стрелка или простая иконка. Не нужно писать текст Свернуть меню внутри рабочей области или в самой кнопке. Подсказка через title допустима.

Левое меню компактное, с группой НСИ и пятью разделами из актуального ТЗ:

```text
Обзор объектов
Дерево объектов
Дерево видов объектов
Техкарты
Справочники
```

Старые пункты широкого бюджетного приложения, например Перечень объектов, Помещения, Системы, Переменные, Сотрудники, Бюджетирование и Аналитика, не входят в левое меню этого модуля НСИ. Они могут существовать в другой системе, но не должны подменять текущую навигацию НСИ.

Рабочая область зависит от раздела:

```text
Обзор объектов -> полноэкранный обзор корневых объектов без правой карточки
Дерево объектов -> дерево слева, карточка справа
Дерево видов объектов -> дерево видов слева, карточка вида справа
Техкарты -> дерево или список техкарт слева, карточка техкарты справа
Справочники -> дерево или список справочников слева, карточка записи справа
```

Для разделов с деревом центральная область дерева должна быть отдельной белой панелью с рамкой, заголовком раздела, поиском, сортировкой и меню действий. Дерево должно занимать фиксированную левую часть рабочей области и иметь собственный скролл. Правая карточка должна занимать основную ширину и иметь собственный контентный скролл при большом объеме данных.

В дереве используются следующие правила:

```text
активная строка подсвечивается мягким голубым фоном
вложенность показывается отступами и вертикальными направляющими
для раскрытия используются компактные стрелки
для типов элементов используются простые пиктограммы
поиск расположен над деревом
сортировка расположена рядом с поиском
действия Создать, Удалить, Копировать доступны в нижней панели или через меню строки
```

Для дерева объектов и дерева видов объектов нижняя панель действий допускается в стиле референса:

```text
Создать
Удалить
Копировать
```

При этом критичные действия не выполняются сразу. Удаление или снятие с учета всегда требует предупреждения в правой карточке.

Правая карточка выбранного элемента должна быть визуально похожа на форму редактирования из референса:

```text
заголовок карточки
вкладки под заголовком
основной блок параметров с таблицей или сеткой полей
дополнительный блок групп параметров ниже
кнопки действия в правой нижней части или в верхней зоне карточки
```

Для объектов во вкладке Параметры форма должна поддерживать режим просмотра и режим редактирования. В режиме редактирования поля должны выглядеть как input/select/checkbox. В режиме просмотра допускается табличное отображение label -> value. Обязательные поля отмечаются визуально, пустые поля показываются только при включенном переключателе Показать пустые.

Для дерева видов объектов карточка должна иметь табличный блок основных свойств и блок групп параметров. Слева внутри блока групп отображается список групп параметров, справа список параметров выбранной группы. Это соответствует референсу, где группы параметров и параметры показаны двумя соседними панелями.

Для технологических карт карточка должна использовать отдельный паттерн вкладок:

```text
Параметры
Операции
Персонал
Материалы
СИЗ
```

Во вкладке Параметры техкарты основные реквизиты отображаются плотной таблицей или формой label -> value. Должны быть видны поля: идентификатор, наименование, тип, вид объекта, вид работ, дата ввода, дата вывода, периодичность, минимальный интервал выполнения, активная, комплексная.

Во вкладке Операции используется табличный или карточный список этапов. Для каждой операции должны быть видны этап, наименование, количество исполнителей или работников, длительность и возможность раскрыть назначенных исполнителей. Внизу можно показывать рассчитанную общую длительность и трудозатраты.

Во вкладке Персонал используется строковая форма с полями исполнитель, разряд, квалификация или электробезопасность, количество, ставка или стоимость, этапы. Если часть справочников еще не реализована, поле может быть временно select/input, но интерфейс не должен быть завязан на конкретный объект вроде ЖК.

Во вкладке Материалы используется повторяемая строка материала с полями наименование, количество, единица измерения, цена, сумма. Сумма считается как количество умножить на цену. Внизу показывается итог по материалам.

Во вкладке СИЗ используется такой же паттерн повторяемых строк, как для материалов: наименование, количество, единица измерения, обязательность, комментарий. Если СИЗ не требуются, пользователь должен видеть осознанно пустой список.

Для правой карточки допустимы кнопки:

```text
Редактировать
Сохранить
Отмена
Сменить вид
Вывести из эксплуатации
Создать техкарту
```

Набор кнопок зависит от раздела и состояния элемента. Например, для объекта может быть Сменить вид, для техкарты Сохранить изменения, для активного объекта или техкарты Вывести из эксплуатации. Красная кнопка используется только для опасного действия.

Интерфейс должен быть спокойным и минималистичным: белые панели, тонкие серые границы, мягкая голубая подсветка активного элемента, зеленые статусы для успешных признаков, красный цвет только для опасных действий и критичных предупреждений. Не нужен визуальный светофор с большим количеством ярких цветов.

Стилистика должна быть ближе к АСКЭ: плотный рабочий интерфейс, аккуратные панели, минимум декоративного текста, минимум поясняющих подписей, больше смысла через структуру, иконки, заголовки и состояние раскрытия. Интерфейс не должен выглядеть как лендинг или презентационная витрина.

На широком экране левое меню, дерево и карточка отображаются рядом. На tablet-ширинах левое меню все равно остается слева, но может быть свернуто до узкой панели. На узких phone-ширинах допускается drawer-режим левого меню, но логика разделов и карточек не меняется.

## 2.2. Обзор объектов

Обзор объектов — начальный раздел НСИ перед Деревом объектов. Его задача — быстро показать, какие основные объекты заведены в НСИ, без раскрытия всей глубокой иерархии.

Обзор объектов не заменяет Дерево объектов. Полная структура, перенос, редактирование вложенности и техническая иерархия остаются в разделе Дерево объектов.

В Обзоре объектов не должно быть правой карточки выбранного элемента. Экран Обзор объектов занимает всю рабочую область справа от левого меню. Нельзя показывать рядом с обзором стандартный DetailsPanel, карточку объекта, карточку системы, карточку оборудования или заглушку карточки.

Обзор объектов должен быть интуитивным и компактным. Не нужно много описательных подписей, длинных пояснений, вводных текстов и рекламных заголовков. Допустимы короткий заголовок раздела, компактная панель действий и сами обзорные деревья.

Главный вид Обзора объектов — одно или несколько свернутых обзорных деревьев корневых объектов. Если в НСИ заведен один корневой объект, показывается одно дерево на всю ширину рабочей области. Если заведено несколько корневых объектов, показываются несколько отдельных деревьев или карточек-деревьев.

Каждое обзорное дерево изначально свернуто. Пользователь сам раскрывает корневой объект и уровни ниже. Состояние раскрытия должно быть локальным для Обзора объектов и не должно ломать раскрытие в полном Дереве объектов.

В свернутой строке корневого объекта показывать только ключевые сведения:

```text
название
вид объекта
уровень детализации
площадь, если есть
помещения
системы
оборудование
техкарты
предупреждения, если есть
```

Не нужно выводить отдельные большие карточки метрик с подписями для каждого показателя. Сводку лучше делать компактной строкой или бейджами в строке дерева.

При раскрытии корневого объекта показывается краткая структура только до уровня детализации. Ниже уровня детализации полная иерархия не разворачивается в Обзоре объектов.

Если уровень детализации равен 2, то для корневого объекта показываются дочерние элементы второго уровня, и на них выводятся компактные блоки:

```text
Помещения
Системы
```

Если уровень детализации равен 3, то сначала показывается путь до третьего уровня, и блоки Помещения и Системы отображаются на элементах третьего уровня.

Блоки Помещения и Системы должны быть компактными. Не надо делать большие информационные карточки с длинными пояснениями. Достаточно названия блока, количества и короткого списка элементов или групп.

В блоке Помещения показываются сгруппированные помещения или зоны: тип, количество и площадь, если есть. В блоке Системы показываются связанные системы, которые применимы на этом уровне детализации. Глобальные системы, привязанные к верхнему уровню, не дублируются вниз, но могут быть показаны как применимые к выбранной области.

В Обзоре объектов нужны действия:

```text
Добавить объект
Создать из шаблона
Открыть в дереве
```

Действия должны быть компактными: в верхней панели обзора или в меню строки. Действие Открыть в дереве переводит пользователя в Дерево объектов и выбирает соответствующий объект.

Запрещено превращать Обзор объектов в экран с двумя колонками Обзор + Карточка. Запрещено автоматически раскрывать правую карточку при выборе строки в Обзоре объектов. Если пользователю нужно редактирование, он переходит через Открыть в дереве.

## 3. Дерево объектов

Дерево объектов описывает реальную структуру объекта инфраструктуры. Ограничений по глубине быть не должно.

Примеры шаблонов:

```text
БЦ -> этаж -> офис -> помещение
ЖК -> дом -> этаж -> помещение
Отель -> корпус -> этаж -> номер
Склад -> зона -> участок -> оборудование
```

Это только шаблоны. Пользователь может создать любую вложенность.

Каждый элемент дерева считается объектом учета. Это может быть здание, этаж, офис, помещение, зона, система, часть системы или оборудование, если пользователь решил вести структуру таким образом.

Офис тоже может быть помещением. Перечень объектов фактически опирается на иерархию помещений и объектов учета.

## 4. Поведение дерева объектов

Дерево по умолчанию свернуто.

В строке элемента показывается краткая сводка: площадь, количество дочерних элементов, количество систем, количество оборудования, количество техкарт, наличие предупреждений.

При выборе элемента справа открывается карточка.

Для выбранного элемента доступны действия: Добавить, Редактировать, Перенести, Снять с учета, Копировать.

Кнопка Добавить открывает список: корневой объект, дочерний объект, помещение, система, оборудование. Корневой объект создается без родителя, дочерние элементы создаются от выбранного объекта.

Перенос работает через drag and drop и через действие Перенести с выбором нового родителя.

Нельзя переносить элемент внутрь самого себя или внутрь собственного дочернего элемента.

При снятии элемента с учета нужно показать предупреждение: сколько внутри дочерних элементов, систем, оборудования, техкарт и какие связи будут затронуты. Пользователь может отменить действие или подтвердить его.

## 5. Дерево видов объектов

Дерево видов объектов является справочником типов и шаблонов. Оно определяет, какие виды объектов можно создавать, какие параметры у них есть и какие дочерние виды типично применяются.

Примеры видов: Здание, Этаж, Офис, Помещение, Система, Оборудование, Датчик, Извещатель, Лифт, Паркинг, Техническое помещение.

Дерево видов не должно жестко ограничивать пользователя. Оно используется как справочник и шаблон. Если нужного вида нет, пользователь может создать новый вид прямо при добавлении объекта.

Для каждого вида хранить: наименование, код, сокращение, иконку, признаки разрешения создания/редактирования/снятия объектов с учета, дочерние виды, группы параметров, параметры, единицы измерения, правила отображения параметров в карточке.

## 6. Шаблоны быстрого старта

При создании корневого объекта пользователь выбирает способ создания: пустой объект или объект из шаблона.

Шаблоны должны быть универсальными. Минимальный набор: Бизнес-центр, Жилой комплекс, Отель, Склад, Производственный объект.

ЖК является только одним из шаблонов, а не базовой моделью системы.

Шаблон должен разворачиваться в реальные объекты учета: корневой объект, дочерние уровни, помещения или зоны. Узлам шаблона назначаются виды объектов, количество, единицы измерения и параметры.

Если у узла шаблона количество больше одного, система создает несколько объектов с понятным числовым суффиксом.

При создании корневого объекта пользователь задает уровень детализации. Этот уровень определяет, начиная с какого уровня дерева должны отображаться и привязываться детальные сущности, например помещения, системы и оборудование.

Цель шаблонов: уменьшить ручное заполнение.

## 7. Карточка выбранного элемента

Справа отображается карточка выбранного элемента.

Во вкладке Параметры для объекта нужны поля: идентификатор, наименование, сокращение, вид объекта, родительский объект, площадь, количество, единица измерения, статус, уровень объекта, уровень детализации, статус доступности детализации, связанные системы, связанное оборудование, связанные техкарты, параметры по виду объекта.

Параметры группируются. Нужно сделать блок групп параметров: слева список групп, справа параметры выбранной группы.

Нужен переключатель Показать пустые.

Если элемент свернут в дереве, ключевые параметры все равно должны быть видны в строке дерева. Особенно важны площадь, предупреждения и параметры вида с признаком showInTree.

## 8. Помещения

Помещение является объектом учета. Офис также может считаться помещением.

Поля помещения: наименование, тип, площадь, количество, единица измерения, родительский объект, связанные системы, связанное оборудование, параметры по виду помещения.

Типы помещений берутся из дерева видов объектов. Если нужного типа нет, пользователь может создать новый тип.

При редактировании помещения связанные системы и оборудование выбираются чекбоксами в правой карточке.

Для списков выбора нужны поиск, чекбоксы, выбрать все, снять все, массовая привязка.

## 9. Системы

Система является отдельной сущностью. Система может содержать вложенные подсистемы и оборудование.

Пример:

```text
Пожарная сигнализация
└── Извещатели / Оповещатели
    └── Датчики
```

Система может быть привязана к любому уровню дерева объектов: всему зданию, этажу, офису, помещению или другой выбранной области.

Если система привязана к верхнему уровню, она не дублируется на нижних уровнях. Она считается глобальной для выбранной области. На нижних уровнях она видна в контексте привязки оборудования или при просмотре связанных систем.

Поля системы: наименование, тип, область применения, связанные узлы дерева, связанные помещения, оборудование, количество, единица измерения, параметры по виду системы.

Область применения: весь объект, конкретный узел дерева, одно помещение, несколько помещений.

При редактировании системы пользователь выбирает связанные узлы и помещения чекбоксами в правой карточке.

## 10. Оборудование

Оборудование ведется внутри системы. Отдельная вкладка оборудования на первом этапе не обязательна.

Оборудование может быть вложенным.

Пример:

```text
Шкаф пожарной автоматики
└── Модуль
    └── Датчик
```

Поля оборудования: наименование, вид оборудования, родительское оборудование, система, помещение или объект размещения, количество, единица измерения, параметры по виду оборудования.

Периодичность обслуживания не хранится в оборудовании. Она относится к технологической карте.

Вид оборудования задается через дерево видов объектов. Единица оборудования обычно одна, но вид оборудования может объединять группу одинаковых элементов.

## 11. Связи

Связи должны быть двусторонними в интерфейсе.

В помещении пользователь видит связанные системы и оборудование. В системе пользователь видит область применения, связанные узлы, связанные помещения и оборудование. В оборудовании пользователь видит систему, место размещения и родительское оборудование.

Нужна массовая привязка: привязать систему ко всем помещениям этажа, привязать систему ко всему объекту, привязать оборудование к выбранным помещениям, привязать техкарту ко всем одинаковым элементам выбранного вида.

## 12. Технологические карты

Техкарты находятся в НСИ, но не формируются автоматически.

Техкарта является нормативной карточкой работ. Она описывает, что нужно выполнить, для какого вида объекта или конкретного объекта применяется работа, какие операции входят в состав, какой персонал нужен, какие материалы и СИЗ требуются.

Карточка технологической карты должна иметь отдельные вкладки:

```text
Параметры
Операции
Персонал
Материалы
СИЗ
```

### 12.1. Вкладка Параметры

Во вкладке Параметры отображаются и редактируются основные реквизиты технологической карты:

```text
Идентификатор
Наименование
Тип
Вид объекта
Вид работ
Дата ввода
Дата вывода
Периодичность
Минимальный интервал выполнения
Активная
Комплексная
```

Назначение полей:

```text
Идентификатор — внутренний id записи.
Наименование — человекочитаемое название техкарты.
Тип — классификация техкарты, например технический осмотр, обслуживание, очистка, проверка.
Вид объекта — вид объекта, системы или оборудования, к которому применима техкарта. Значение берется из дерева видов объектов.
Вид работ — справочник видов работ.
Дата ввода — дата и время начала действия техкарты.
Дата вывода — дата и время окончания действия техкарты. Если не задана, техкарта считается действующей при активном статусе.
Периодичность — нормативная частота выполнения работы.
Минимальный интервал выполнения — минимально допустимый промежуток между повторными выполнениями одной и той же работы.
Активная — признак доступности техкарты для использования.
Комплексная — признак, что техкарта объединяет несколько операций или работ в один комплекс.
```

Поле Вид объекта должно быть ссылочным. Пользователь выбирает значение из дерева видов объектов, например помещение, серверный шкаф, насос, вентиляционная установка, система электроснабжения.

Поле Вид работ должно браться из справочника видов работ. Если нужного вида работ нет, пользователь должен иметь возможность добавить его в справочник.

Периодичность и минимальный интервал выполнения могут быть не заданы на черновом этапе, но такая техкарта должна подсвечиваться предупреждением до ввода в активное использование.

### 12.2. Вкладка Операции

Во вкладке Операции пользователь задает состав работ.

Каждая операция должна хранить:

```text
номер операции
наименование операции
описание или инструкцию выполнения
признак обязательности
порядок выполнения
ожидаемый результат
```

Минимальное требование: у активной техкарты должна быть хотя бы одна операция.

Операции должны редактироваться в карточке техкарты, без отдельного внешнего документа.

### 12.3. Вкладка Персонал

Во вкладке Персонал указываются требования к исполнителям.

Для каждой строки персонала хранить:

```text
должность
квалификация
разряд
количество исполнителей
минимальная длительность, чел-ч
комментарий
```

Должности, квалификации и разряды должны браться из справочников.

Если техкарта активная, но персонал не задан, нужно показывать предупреждение.

### 12.4. Вкладка Материалы

Во вкладке Материалы задается перечень материалов и расходников.

Для каждой строки материала хранить:

```text
материал
количество
единица измерения
обязательность
комментарий
```

Материалы должны браться из справочника материалов. Если материалы для техкарты не требуются, вкладка может быть пустой, но пользователь должен понимать, что это осознанно пустой список.

### 12.5. Вкладка СИЗ

Во вкладке СИЗ задается перечень средств индивидуальной защиты.

Для каждой строки СИЗ хранить:

```text
наименование СИЗ
количество
единица измерения
обязательность
комментарий
```

СИЗ должны браться из справочника СИЗ.

### 12.6. Привязка техкарты

Техкарта должна быть привязана минимум к одному применимому контексту:

```text
вид объекта
конкретное помещение
конкретная система
конкретное оборудование
```

Базовый сценарий карточки — привязка через поле Вид объекта. Это позволяет создать одну техкарту для всех объектов одинакового вида, например для всех серверных шкафов.

При необходимости техкарта может уточняться конкретным объектом применения: помещением, системой или оборудованием.

Техкарта не должна существовать как независимая запись без области применения.

### 12.7. Обязательные поля и предупреждения

Для активной техкарты обязательны:

```text
наименование
тип
вид объекта или конкретный объект применения
вид работ
дата ввода
периодичность или минимальный интервал выполнения
минимум одна операция
персонал, если работа требует исполнителя
```

Если обязательные поля не заполнены, интерфейс должен показывать предупреждения прямо в карточке.

Дата вывода не обязательна. Если дата вывода наступила, техкарта считается выведенной из действия и не должна предлагаться для новых работ.

Планирование работ, расчет объемов и формирование графика выполнения не относятся к НСИ. НСИ хранит только нормативную техкарту и связи.

## 13. Справочники

Минимальный состав справочников: виды работ, дефекты, единицы измерения, должности, квалификации, разряды, материалы, СИЗ.

Для карточки техкарты особенно важны справочники: виды работ, должности, квалификации, разряды, материалы, СИЗ, единицы измерения.

Интерфейс справочников такой же: слева дерево или список, справа карточка выбранного элемента с параметрами и записями.

## 14. Предупреждения

Отдельная вкладка проверки НСИ не нужна. Ошибки и предупреждения нужно подсвечивать прямо при заполнении объектов и карточек.

Подсвечивать нужно все, что может сломать дальнейшую логику: объект без вида, помещение без родителя, система без области применения, оборудование без системы, оборудование без места размещения, техкарта без вида объекта или объекта применения, техкарта без вида работ, техкарта без даты ввода, техкарта без периода или минимального интервала, техкарта без операций, активная техкарта без персонала, действие над элементом с вложенными связями.

Процент заполненности не нужен.

## 15. Тестовые данные

Нужен включаемый режим тестовых данных.

Переключатель тестовых данных должен находиться в верхней служебной панели. Он должен быть компактным и понятным: например Тестовые данные или Демо. Переключатель не должен занимать левое меню и не должен быть отдельным разделом.

Когда тестовые данные выключены, пользователь видит пустой НСИ и может заполнить его с нуля. В этом состоянии не нужно заставлять пользователя выбирать тип объекта или шаблон до первого действия.

Когда тестовые данные включены, загружается реалистичный демо-набор. Тестовые данные должны быть редактируемыми как обычные данные.

Основной демо-объект: БЦ Lucky.

Демо БЦ Lucky должно быть достаточно реалистичным, чтобы по нему можно было нормально проверять Обзор объектов, Дерево объектов, системы, оборудование, техкарты и связи.

Минимальная структура БЦ Lucky:

```text
БЦ Lucky
├── -1 этаж / паркинг и технические зоны
├── 1 этаж / входная группа, ресепшен, коммерческие помещения, МОП
├── 2 этаж / офисные блоки и переговорные
├── 3 этаж / офисные блоки и сервисные помещения
├── 4 этаж / Офис СОТА и связанные помещения
├── 5 этаж / офисные блоки
├── 6 этаж / офисные блоки
└── 7 этаж / технические помещения и инженерные зоны
```

Каждый этаж должен быть представлен как отдельный объект учета. На каждом этаже должны быть дочерние помещения или зоны. Не надо оставлять этажи пустыми.

Для 4 этажа обязательно добавить Офис СОТА как отдельный объект уровня детализации. Внутри Офиса СОТА добавить реалистичные помещения:

```text
open space
переговорная
кабинет руководителя
серверная
кухня / кофе-поинт
санузел
кладовая / подсобное помещение
коридор / МОП
```

Для остальных этажей добавить по несколько реалистичных помещений или офисных блоков, например офисный блок A, офисный блок B, переговорная, санузел, коридор, техническое помещение, кладовая, электрощитовая.

Рекомендуемый уровень детализации для БЦ Lucky: 3.

Это означает:

```text
1 уровень: БЦ Lucky
2 уровень: этаж
3 уровень: офис, зона или крупный блок на этаже
```

В Обзоре объектов для БЦ Lucky должны быть видны этажи, а при раскрытии этажей — объекты уровня детализации и компактные блоки Помещения и Системы. Полную глубину помещений ниже уровня детализации раскрывать только в Дереве объектов.

Для БЦ Lucky добавить реалистичные системы:

```text
электроснабжение
освещение
отопление
водоснабжение
канализация
вентиляция и кондиционирование
пожарная сигнализация
СКУД
видеонаблюдение
лифтовое оборудование
слаботочные сети / интернет
```

Системы должны быть привязаны к корректным уровням дерева. Глобальные системы привязывать к БЦ Lucky или этажу, локальные системы — к офису, помещению или зоне. Нельзя дублировать глобальные системы на каждом нижнем уровне.

Для БЦ Lucky добавить оборудование внутри систем, например электрощит, серверный шкаф, вентиляционная установка, кондиционер, камера, контроллер СКУД, пожарный извещатель, насос, лифтовой шкаф. Оборудование должно иметь место размещения.

Для БЦ Lucky добавить несколько техкарт, привязанных к системам и оборудованию, например чистка серверного шкафа, осмотр вентиляционной установки, проверка пожарной сигнализации, осмотр электрощита, проверка СКУД.

Кроме БЦ Lucky можно оставить дополнительные демо-объекты: ЖК, Отель, Склад, Производственный объект. Но они не должны быть пустыми заглушками. Если времени мало, главным полным демо-набором должен быть БЦ Lucky.

Демо-данные должны быть достаточно полными, примерно на 8 из 10: многоуровневая иерархия, помещения, глобальные системы, локальные системы, оборудование, вложенное оборудование, техкарты, справочники.

Демо-техкарты должны показывать разные сценарии: активная простая техкарта, комплексная техкарта, техкарта без материалов, техкарта с СИЗ, техкарта с датой вывода.

{

## 17. Этапы разработки

1. Каркас: разделы, левое меню, центральное дерево и правая карточка.
2. Дерево видов объектов: справочник видов, параметры, группы, единицы измерения.
3. Дерево объектов: создание, редактирование, перенос, копирование, сводка в строке, drag and drop.
4. Шаблоны: БЦ, ЖК, Отель, Склад, Производственный объект, уровень детализации.
5. Техкарты: карточка с вкладками Параметры, Операции, Персонал, Материалы, СИЗ; привязка к виду объекта, помещению, системе или оборудованию.
6. Стабилизация интерфейса по референс-скринам: верхняя панель, левое сворачиваемое меню, дерево, правая карточка, табличные формы, нижние действия, единый визуальный стиль.
7. Обзор объектов: начальная сводная вкладка перед деревом объектов, полноэкранный обзор без правой карточки, свернутые обзорные деревья корневых объектов, краткое отображение до уровня детализации, блоки Помещения и Системы на уровне детализации.
8. Системы: вложенность, область применения, привязка к любому уровню, чекбоксы, поиск, массовые действия.
9. Оборудование: вложенность, привязка к системе, размещение, количество, единицы, параметры.
10. Связи: двусторонние связи между помещениями, системами, оборудованием и объектами учета.
11. Справочники: виды работ, дефекты, единицы измерения, должности, квалификации, разряды, материалы, СИЗ.
12. Предупреждения: подсветка критичных проблем.
13. Демо-данные: БЦ Lucky как основной реалистичный демо-набор, дополнительные объекты при необходимости.

## 18. Что нельзя делать

- Нельзя привязывать архитектуру только к ЖК.
- Нельзя использовать ЖК или complex как базовую сущность модели.
- Нельзя делать системы строками внутри помещения.
- Нельзя дублировать глобальную систему на всех нижних уровнях дерева.
- Нельзя делать техкарты независимыми от объектов учета или вида объекта.
- Нельзя делать массовое редактирование через неудобный multiple select.
- Нельзя выполнять критичные действия без предупреждения о вложенных данных и связях.
- Нельзя копировать содержимое деревьев со скриншотов.
- Нельзя переносить в НСИ старую навигацию широкого бюджетного приложения вместо текущих разделов НСИ.
- Нельзя переносить левое меню НСИ наверх на tablet и desktop-ширинах. Левое меню должно оставаться слева и сворачиваться.
- Нельзя превращать Обзор объектов в полное Дерево объектов. Обзор показывает только краткую структуру до уровня детализации.
- Нельзя показывать правую карточку выбранного элемента на начальном экране Обзор объектов.
- Нельзя перегружать Обзор объектов длинными пояснениями, большими метриками и лишними подписями. Это должен быть компактный рабочий экран в стилистике АСКЭ.
- Нельзя оставлять главный демо-объект пустой заглушкой. БЦ Lucky должен иметь этажи, помещения, системы, оборудование и техкарты.
- Нельзя писать текст Свернуть меню в кнопке сворачивания бокового меню. Должна быть только стрелка или компактная иконка.

## 19. Главный результат

Пользователь должен иметь возможность открыть пустой НСИ, включить реалистичные тестовые данные сверху страницы, увидеть компактный сводный обзор объектов, создать объект любого типа, выбрать или создать вид объекта, применить шаблон, отредактировать полное дерево, создать помещения, создать глобальные и локальные системы, добавить оборудование, связать данные, создать техкарты с операциями, персоналом, материалами и СИЗ и использовать НСИ как исходную базу для других модулей.
